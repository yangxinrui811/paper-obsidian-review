# Paper Obsidian Review

## 个性化版本说明

本版本在原始 paper-obsidian-review 技能基础上，增加了以下个性化能力：
- 统一文件命名规范（年份-期刊缩写-短语描述）
- 多课题 Obsidian vault 分子文件夹组织
- PDF 文件与笔记文件的命名一致性和互链管理
- 引文网络分析——从已有文献中反向挖掘缺失的奠基性论文
- 批量论文下载与 DOI 验证（Semantic Scholar API / CrossRef / arXiv）
- 多课题文献总览笔记自动生成

## 目标

把论文、代码仓库、本地 PDF 和在线资料整理成可直接放入 Obsidian 的中文论文调研笔记。支持多课题并行管理、文献库补齐、批量下载。输出应适合后续写项目依据、文献综述、复现计划和指标支撑材料。

核心原则：不要写泛泛的"这篇工作干了什么"。必须明确"为什么做、解决什么问题、用什么方法解决、创新在哪里、如何复现、与用户指标如何关联"。

## 触发场景

使用本 skill 当用户要求：

- 总结论文、代码、相关工作、论文调研、文献综述。
- 把论文内容放进 Obsidian、vault、Web Clipper、Markdown 笔记。
- 需要整理本地 PDF、本地代码、在线论文、在线代码链接。
- 需要提取论文方法图、框架图、系统图。
- 需要判断代码复现可行性。
- 需要把多篇论文按统一结构对比。
- 需要从已有文献的引用网络中找出缺失的关键论文。
- 需要批量下载论文并统一管理 DOI。

## 输入信息收集

优先向用户确认或从本地发现以下信息：

- Obsidian vault 路径，例如 `E:\obsidian\human-robot`。
- 本地代码根目录，例如 `C:\code\human-robot`。
- 本地 PDF 目录，例如 `D:\论文\人机交互`。
- 每篇论文的在线论文链接、在线代码链接、项目主页。
- 用户关心的指标或主题，例如"大模型驱动的人机混合增强智能""多模态人机交互"。

如果用户未给 vault 路径，先寻找 `.obsidian` 文件夹；如果找不到，指导用户在 Obsidian 主程序中创建/打开 vault。不要把浏览器插件 Obsidian Web Clipper 当作 Obsidian 主程序。

## 推荐工作流

### 1. 盘点材料

- 列出 PDF 目录中的 PDF 文件名。
- 列出代码目录中的项目名、README、依赖文件、入口脚本、图片文件。
- 对每篇论文建立映射：论文名 -> 本地 PDF -> 本地代码 -> 在线论文 -> 在线代码。
- 对不确定映射明确标注"待确认"，不要猜错链接。

### 2. 统一文件命名

**PDF 和笔记文件统一使用以下命名格式：**

```text
YYYY-JournalAbbrev-短语描述.pdf
YYYY-JournalAbbrev-短语描述.md
```

命名规则：
- 年份在前，便于按时间排序
- 期刊缩写紧跟年份（如 PRL, JACS, JCTC, NatCommun, Science, PNAS）
- 短语描述用中文，突出论文的核心贡献/方法/平台名称
- PDF 文件名和笔记文件名保持一致（仅扩展名不同）

示例：
- `2008-PRL-WTMetaD收敛可调自由能方法.pdf` / `.md`
- `2020-PRX-OPES在线概率增强采样.pdf` / `.md`
- `2015-PNAS-隧穿无大KIE.pdf` / `.md`

### 3. 建立 Obsidian 结构

**多课题 vault 组织方式：**

```text
<vault>/
  pdfs/              ← 所有课题共享的 PDF
  attachments/       ← 所有课题共享的图片
  课题名称1/          ← 课题1的笔记（子文件夹）
    课题1-文献总览.md
    2008-PRL-WTMetaD收敛可调自由能方法.md
    ...
  课题名称2/          ← 课题2的笔记
    课题2-文献总览.md
    ...
```

- `pdfs/` 和 `attachments/` 放在 vault 根目录，所有课题共享
- 每个课题的笔记放在独立子文件夹中
- 总览笔记命名为 `课题名称-文献总览.md` 放在对应子文件夹内
- PDF 链接使用 vault 相对路径 `[[pdfs/xxx.pdf]]`，不受子文件夹影响

### 4. 处理 PDF 链接

优先把 PDF 复制到 vault 内部 `pdfs/` 文件夹，然后使用 Obsidian 内部链接：

```markdown
- 本地 PDF：[[pdfs/文件名.pdf]]
```

不要优先使用绝对路径 Markdown 链接，因为 `D:/中文路径/带空格.pdf` 在 Obsidian 中可能无法直接打开。

### 5. 处理论文方法/框架图

只保留"论文方法图、框架图、系统图、pipeline 图"。不要把统计图、结果图、环境演示图、导航示例图当作方法图，除非用户明确要求。

推荐放入：

```text
<vault>/attachments/<paper>-method.png
```

并在笔记中引用：

```markdown
## 论文方法/框架图

![[attachments/<paper>-method.png]]
```

若本地代码没有图片，可从 README、项目主页或论文 PDF 中提取。若无法确认是哪张方法图，先说明候选图并让用户确认。

### 6. 单篇笔记结构

每篇论文使用以下结构。标题必须学术化、清晰，不要口语化。

```markdown
---
title: "..."
source: "..."
DOI: "..."
code: "..."
tags:
  - paper
  - ...
created: YYYY-MM-DD
---

# 论文简称 / 系统名

## 基本信息

论文题目：...

## 链接

- 本地 PDF：[[pdfs/xxx.pdf]]
- 本地代码：`...`
- 在线论文：https://doi.org/...
- 在线代码：...
- 项目主页：...

## 论文方法/框架图

![[attachments/xxx-method.png]]

## 研究背景与问题动因

说明该方向为什么需要这篇工作。写研究动因，不写泛泛宣传。

## 核心科学问题

用"核心问题 + 子问题"写清楚论文要解决什么。

## 问题求解路径

逐条写明：论文使用什么方法，解决什么问题。
格式推荐：

- 使用 X，解决 Y 问题。
- 使用 A+B 融合，解决 C 的不确定性问题。

## 方法定位

说明该方法属于什么范式、系统、任务或模型类型，在相关研究链条中的位置是什么。

## 输入输出

输入：...

输出：...

## 方法框架

按模块写清楚系统组成和数据流。

## 关键机制

提炼论文真正起作用的机制。

## 主要创新贡献

列出 3-6 条，强调相对已有工作的新增点。

## 数据集设计 / 代码结构 / 配置依赖

按论文类型选择一个标题。

## 复现要点

说明入口脚本、运行方式、环境依赖、硬件要求和主要卡点。

## 与指标关联

把论文明确对应到用户的项目指标、课题指标或研究主线。
```

### 7. 引文网络分析与文献补齐

当用户已有充分的文献集但需要发现遗漏的重要论文时，执行以下流程：

#### 7.1 提取参考文献

- 优先处理综述论文和近期高影响力论文的引文
- 使用 pdfplumber 提取每篇论文的最后 25-30% 页面（参考文献通常在后部）
- 关注引言部分中反复出现的论文引用

#### 7.2 识别奠基性论文

基于以下因素判断论文是否为"缺漏的奠基性文献"：
- 在领域综述中被反复引用
- 是某个方法/理论的原始论文
- 是领域标准 benchmark 的来源
- 被已有文献的引言部分多次列为"prior work"

#### 7.3 交叉比对

- 将候选论文与用户已有的 PDF 目录进行比对
- 排除已存在于任何子文件夹中的论文
- 按紧急程度排序输出缺失清单

#### 7.4 输出格式

按课题分组，输出表格：

```markdown
| 年份 | 期刊 | 作者 | 描述 | 完整引用 |
|---|---|---|---|---|
```

### 8. 论文下载与 DOI 管理

#### 8.1 DOI 查询与验证

- 使用 CrossRef API (`https://api.crossref.org/works/{DOI}`) 验证 DOI 正确性
- 使用 Semantic Scholar API 搜索论文标题获取 DOI
- 修复常见 DOI 错误（如多位数、错位字符）

#### 8.2 论文下载策略

按优先级尝试：
1. **Semantic Scholar API** (`api.semanticscholar.org`)：查找 openAccessPdf 链接
2. **arXiv**：搜索预印本版本
3. **PubMed Central**：NIH 资助论文的免费版本
4. **出版商 OA**：开放获取期刊的直接链接
5. **Sci-Hub**：仅在前述方法都失败时尝试（可能被防火墙屏蔽）

#### 8.3 下载后处理

- 将下载的 PDF 重命名为统一格式
- 复制到 vault 的 `pdfs/` 目录
- 删除非出版版本的预印本（如 arXiv 版本），让用户自行下载正式出版版本以便规范引用
- 已存在于 vault 中的论文自动跳过

### 9. 多课题总览笔记

创建一篇总览笔记，用于横向比较：

```markdown
# 多篇论文对比

## 对比表

| 工作 | 主要对象 | 输入模态 | 输出/目标 | 适合支撑的研究点 | 复现难度 |
|---|---|---|---|---|---|

## 链接汇总

| 工作 | 本地笔记 | 本地 PDF | 本地代码 | 在线论文 | 在线代码 |
|---|---|---|---|---|---|

## 总体理解

用一段话串起多篇工作的研究链条。

## 与项目指标的对应关系

分指标说明哪篇最直接、哪篇间接支撑。

## 复现优先级建议

给出先复现哪篇、为什么、卡点是什么。
```

对于**课题级总览**，额外包含：

#### 文献演化脉络

将课题内论文按时间/主题分组，形成"奠基期→拓展期→现代"的演化时间线：

```markdown
## 文献演化脉络

### 奠基期 (XXXX-XXXX)
| 论文 | 贡献 |
|---|---|

### 拓展期 (XXXX-XXXX)
...
```

#### 课题定位建议
结合文献集为用户的课题提供定位分析（问题→方法→验证→对标）

## 复现可行性判断

评估复现时至少检查：

- README 是否完整。
- 是否有 requirements、environment、Dockerfile、docker-compose。
- 是否需要 ROS、Gazebo、CUDA、特定设备、模型权重、API key。
- 是否有数据集、预训练权重、运行脚本。
- 当前机器/服务器是否具备硬件和系统环境。

## 标题规范

优先使用这些学术化标题：

- `研究背景与问题动因`
- `核心科学问题`
- `问题求解路径`
- `方法定位`
- `输入输出`
- `方法框架`
- `关键机制`
- `主要创新贡献`
- `数据集设计`
- `代码结构`
- `配置依赖`
- `复现要点`
- `与指标关联`

避免：

- `这篇工作干了什么`
- `简单介绍`
- `大概内容`
- `我的理解`，除非用户明确要个人化笔记。

## 常见坑

- 不要把 Web Clipper 插件当成 Obsidian 主程序；插件不能单独作为 vault。
- 不要用绝对本地 PDF 链接作为首选；复制进 vault 后用内部链接。
- 不要擅自创建多层目录；先尊重用户指定的 vault 结构。
- 不要把结果图、统计图当方法框架图。
- 不要猜测 PDF 与论文的对应关系；不确定就标待确认。
- 不要只写动机或摘要；必须写清"方法解决问题"的映射。
- Windows 路径中有中文或空格时，Obsidian 内部链接比 Markdown 绝对链接更稳。

## 文件管理规范

### 链接更新

- 当 PDF 或笔记文件重命名时，必须同步更新所有 `[[wiki链接]]`
- 使用 Python 脚本批量扫描并替换旧链接为新链接
- 优先更新总览笔记中的链接汇总表

### 文件去重

- 检查并删除压缩副本（如 `_compressed.pdf`）
- 同一篇论文的补充材料标记为 `-SM.pdf`，与主论文放在同一目录

### 源目录与 Vault 同步

- 源目录中的 PDF 改名后，同步复制到 vault 的 `pdfs/`
- vault 的 `pdfs/` 是所有课题共享的，不要按课题建子文件夹

## 输出检查

完成前检查以下全部：

- 三类链接都能在笔记中看到：本地 PDF、本地代码、在线论文/代码。
- PDF 已在 `pdfs/` 下，图片已在 `attachments/` 下。
- 每篇都有方法/框架图，且不是统计/结果图。
- 每篇都有"研究背景与问题动因、核心科学问题、问题求解路径、主要创新贡献"。
- 每篇保留"方法定位、输入输出、方法框架、关键机制、复现要点、与指标关联"。
- 总览表中的链接和单篇笔记保持一致。
- **新增**：PDF 文件名和笔记文件名符合 `YYYY-JournalAbbrev-短语描述` 格式。
- **新增**：论文已按课题分入独立子文件夹，`pdfs/` 和 `attachments/` 在 vault 根目录共享。
- **新增**：如有从引文网络新发现的缺失论文，已输出 DOI 清单并指导用户下载。
- **新增**：下载的论文已经过 DOI 验证（CrossRef），预印本已替换为正式版本。
- **新增**：所有内部 `[[wiki链接]]` 在重命名后已同步更新。

## 工具依赖

- `pdfplumber`：PDF 文本提取和方法图导出
- `urllib` + Semantic Scholar API（需要 API key `s2k-xxx`）：论文搜索和 OA 下载
- CrossRef API：DOI 验证
- `os` / `shutil`：文件批量改名、复制、链接更新
